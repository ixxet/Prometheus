# ADR 0001: `v0.4.0` Memory And Archive Boundaries

Date: 2026-03-26

Status: Accepted

## Context

`v0.3.0` established the first agent runtime:

- `LangGraph` is live
- `Postgres` is the only durable execution store
- `vLLM` is the only model backend

The next milestone needs two new capabilities:

1. semantic memory across runs
2. human-readable summaries and ADR exports

There are real constraints:

- stay self-hosted and OSS for the core runtime path
- avoid paid or licensed dependencies in the first memory/archive iteration
- do not introduce overlapping memory systems
- keep `Open WebUI` as a frontend, not an orchestrator or memory owner
- keep `Obsidian` outside the cluster; do not turn the vault into a cluster PVC

## Decision

### 1. Semantic memory system

Use **Mem0 OSS** as the only semantic-memory target for the `v0.4.0` line.

Do not run:

- `LangMem`
- `Graphiti / Zep`
- `Letta`

alongside it.

`LangMem` remains the documented alternative only.

### 2. Archive sink

Use an **external Obsidian vault sink** for human-readable summaries and ADRs.

That means:

- no Obsidian app in Kubernetes
- no vault PVC in the cluster
- no attempt to make the vault the primary machine-memory store

### 3. Ownership boundary

The **LangGraph runtime** owns both integrations:

- semantic-memory reads and writes
- summary and ADR export

`Open WebUI` stays a UI.

`vLLM` stays a model server.

### 4. Delivery shape

Start `v0.4.0` by adding explicit integration seams inside the LangGraph
service before deploying new stateful components.

This means:

- add a semantic-memory provider interface
- add an archive-sink interface
- default both to no-op implementations until the live services are ready

### 5. Failure policy

Execution state remains the hard requirement.

So:

- `Postgres`/checkpoint persistence failures are run-breaking
- semantic-memory failures should not destroy a successful run
- archive-export failures should be recorded, but should not destroy a
  successful run

## Consequences

- `v0.4.0` will add at least one more internal subsystem beyond Postgres
- the backing vector store and embedding path are separate implementation
  decisions that will follow the seam work
- the repo should show the distinction between:
  - live `v0.3.0` runtime
  - authored `v0.4.0` seams
  - later live memory/archive rollout

## Rejected alternatives

### Run both Mem0 and LangMem

Rejected because it creates overlapping responsibility and unclear truth for
semantic memory.

### Make Obsidian the primary machine-memory store

Rejected because Obsidian is for human-readable artifacts, not checkpoint or
semantic-memory authority.

### Push memory ownership into Open WebUI

Rejected because it couples memory to the chat frontend instead of the agent
runtime.
