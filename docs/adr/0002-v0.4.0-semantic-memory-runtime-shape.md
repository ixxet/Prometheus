# ADR 0002: `v0.4.0` Semantic-Memory Runtime Shape

## Status

Accepted on 2026-03-26.

## Context

`v0.4.0` needs a real semantic-memory system, not just a no-op seam. The main
design choice was where that memory boundary should live:

- a separate Mem0-facing service that LangGraph calls over HTTP
- or a Mem0-backed provider inside LangGraph, with external support services
  only where they add clear value

Prometheus already has a clear execution owner:

- LangGraph owns thread and run state
- Postgres owns checkpoint durability
- `vLLM` owns model serving

Adding semantic memory should not blur those ownership lines.

## Decision

Keep the semantic-memory boundary **inside LangGraph** and back it with:

- **Mem0 OSS** as the semantic-memory library
- **Qdrant** as the vector store
- **TEI** (`text-embeddings-inference`) as the embedding endpoint

Stage `Qdrant + TEI` as a separate suspended GitOps layer under
`homelab-gitops/infrastructure/semantic-memory/`.

Keep the live runtime on:

- `SEMANTIC_MEMORY_PROVIDER=none`

until that support stack is intentionally enabled and validated.

## Why

This keeps the boundaries clean:

- LangGraph still owns orchestration and when memory is consulted
- Mem0 owns memory extraction, search, and update behavior
- Qdrant and TEI stay replaceable support services instead of becoming the
  center of the platform

It also avoids adding a second agent-adjacent application surface before it is
actually needed.

## Consequences

Positive:

- Prometheus gets a real semantic-memory code path now
- the support stack can be activated later without another large refactor
- LangGraph can inject recalled memories explicitly into prompt construction

Tradeoffs:

- the LangGraph image now needs Mem0 and embedding dependencies
- semantic memory is still intentionally staged, not live
- external archive export remains a separate concern

## Rejected Alternatives

### Separate Mem0-first API service as the primary boundary

Rejected because the clean service split was weaker than it looked for this
stack. It would have added another application API surface without improving the
orchestrator boundary.

### Keep semantic memory entirely deferred until archive export is ready

Rejected because it would delay the core `v0.4.0` shape longer than necessary.
The semantic-memory path is the next meaningful runtime capability after
LangGraph persistence.
